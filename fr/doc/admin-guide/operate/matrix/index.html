Matrix et les ponts de messagerie

Activer le serveur Matrix (Synapse), son client web Element et les ponts mautrix (WhatsApp, Signal, Telegram, Messenger, Discord). Pour l’architecture, voir Matrix dans DNF.

Activer Matrix

Section intitulée « Activer Matrix »

  1. Déclarer les services sur l’hôte cible dans etc/config.yaml :

  2. Déclarer l’administrateur Matrix (clé globale, hors hôtes) :

    Ce compte administre les ponts et reçoit les alertes de supervision.

  3. Créer les secrets requis (voir tableau ci-dessous), puis déployer.

L’authentification passe par le SSO Kanidm (OIDC) : tout utilisateur déclaré se connecte avec son compte réseau, aucune création de compte Matrix n’est nécessaire.

Secrets requis

Section intitulée « Secrets requis »

À renseigner dans usr/secrets/secrets.yaml :

Clé sopsRôleGénération
matrix-rss-passwordRegistration shared secret Synapseopenssl rand -hex 32
oidc-secret-matrixSecret du client OIDC Kanidmcf. SSO
turn-secretSecret partagé coturn (si turn activé)openssl rand -hex 32
mautrix-doublepuppet-as-tokenToken appservice double puppeting (commun aux ponts)openssl rand -hex 32
mautrix-doublepuppet-hs-tokenIdem, requis par Synapse, jamais utiliséopenssl rand -hex 32

Gérer les ponts

Section intitulée « Gérer les ponts »

Les ponts WhatsApp, Signal, Telegram et Messenger sont activés par défaut avec Matrix ; Discord est désactivé par défaut. Réglage par hôte dans la configuration nix de la machine :

Chaque pont a ses secrets propres :

PontClés sopsRemarque
WhatsAppmautrix-whatsapp-as-token, mautrix-whatsapp-hs-token, mautrix-whatsapp-encryption-pickle-keyopenssl rand -hex 32 chacun
Signalmautrix-signal-as-token, mautrix-signal-hs-token, mautrix-signal-encryption-pickle-keyopenssl rand -hex 32 chacun
Messengermautrix-meta-as-token, mautrix-meta-hs-token, mautrix-meta-encryption-pickle-keyopenssl rand -hex 32 chacun
Telegrammautrix-telegram-api-id, mautrix-telegram-api-hash, mautrix-telegram-as-token, mautrix-telegram-hs-tokenAPI id/hash à créer sur my.telegram.org 🡕
Discordmautrix-discord-as-token, mautrix-discord-hs-tokenopenssl rand -hex 32 chacun

Les tokens as/hs rendent la registration appservice de chaque pont déterministe (voir les tokens d’appservice).

Migration d'un pont déjà déployé

Sur un pont WhatsApp ou Signal déployé avant l’introduction de ces tokens, l’enregistrement existant conserve les anciens tokens auto-générés. Pour basculer sur les tokens sops : supprimer /var/lib/mautrix-<pont>/<pont>-registration.yaml, redémarrer le pont (l’enregistrement est régénéré depuis la config), puis redémarrer matrix-synapse. Pour Messenger et Discord, un simple redéploiement suivi d’un redémarrage de Synapse suffit : leurs modules réécrivent la registration depuis la config à chaque démarrage.

Permissions

Section intitulée « Permissions »

Rien à gérer : tout compte local (@*:domain.tld) peut utiliser chaque pont avec son propre compte distant, sans interférence entre utilisateurs. Le compte matrix.admin dispose des commandes d’administration des bots (help en conversation directe avec le bot pour la liste).

Double puppeting

Section intitulée « Double puppeting »

Les ponts partagent un appservice doublepuppet (token mautrix-doublepuppet-as-token) : les messages envoyés depuis le téléphone apparaissent dans Matrix comme venant du vrai compte de l’utilisateur. C’est automatique pour tous les comptes locaux, aucune action par utilisateur.

Ouvrir le réseau à la fédération

Section intitulée « Ouvrir le réseau à la fédération »

Par défaut, le serveur fédère avec tous les serveurs Matrix, tout en restant invisible dans les recherches (annuaires non exposés). Pour contrôler la portée, régler darkone.service.matrix.federation dans la config nix de l’hôte :

ObjectifRéglage
Réseau isolé (aucune fédération)enable = false;
Ouvert à tout l’écosystème Matrixenable = true; whitelist = [ ];
Restreint à des serveurs de confianceenable = true; whitelist = [ "ami.org" "matrix.org" ];

Réseau familial

La liste blanche est bidirectionnelle : un serveur non listé ne peut ni écrire à vos utilisateurs ni recevoir d’eux. Pour un réseau utilisé par des enfants, renseigner whitelist plutôt que de laisser la fédération ouverte.

Vérifier après déploiement :

  1. Tester la fédération entrante avec federationtester.matrix.org 🡕 sur domain.tld → tous les voyants verts.

  2. En liste blanche, un serveur non listé est refusé (journal Synapse : Federation denied), un domaine listé fonctionne.

Inscrire des amis (comptes locaux)

Section intitulée « Inscrire des amis (comptes locaux) »

En plus des utilisateurs du SSO, on peut ouvrir une inscription sur jeton pour des comptes locaux (amis). Aucune inscription n’est possible sans jeton.

  1. Activer l’inscription sur jeton dans la config nix de l’hôte, puis déployer :

  2. Promouvoir le compte matrix.admin en server admin (une seule fois), sur l’hôte Matrix :

  3. Créer un jeton d’inscription via l’API d’administration (avec un access token du compte admin) :

    La réponse contient le token à transmettre à l’ami.

  4. L’ami s’inscrit dans Element : Créer un compte → serveur domain.tld → il colle le jeton lorsqu’il est demandé.

L'access token admin

Récupérer un access token dans Element (Paramètres → Aide & à propos → Avancé) avec le compte promu, ou par connexion API (POST /_matrix/client/v3/login). La future interface Synapse-Admin simplifiera promotion et jetons.

Authentification nouvelle génération (MAS)

Section intitulée « Authentification nouvelle génération (MAS) »

Le Matrix Authentication Service 🡕 remplace l’authentification interne de Synapse (requis par Element X). Pour ce qu’il change et comment il s’insère, voir l’architecture.

En cours de validation

Implémenté mais pas encore éprouvé en production : la migration d’un serveur existant (syn2mas) et l’outillage mas-cli sur l’hôte restent à valider. En attendant, laisser mas.enable = false (défaut) : l’authentification OIDC classique reste pleinement fonctionnelle.

  1. Créer les secrets dédiés dans usr/secrets/secrets.yaml :

    Clé sopsRôleGénération
    mas-encryption-secretChiffrement interne MASopenssl rand -hex 32
    mas-synapse-secretSecret partagé MAS ↔ Synapseopenssl rand -hex 32
    mas-rsa-private-keyClé de signature des tokensopenssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:4096

  2. Activer l’option dans la config nix de l’hôte, puis déployer :

Secret immuable

mas-encryption-secret ne doit jamais changer après le premier démarrage de MAS : il protège des données chiffrées en base, qui deviendraient illisibles.

Migrer un serveur déjà en production

Section intitulée « Migrer un serveur déjà en production »

Basculer un Synapse existant exige une fenêtre de maintenance : MAS devient le propriétaire des comptes et sessions, qui doivent être importés avec l’outil officiel syn2mas 🡕 (comptes, mots de passe, sessions, liens OIDC). Sans cette migration, tous les utilisateurs sont déconnectés et les comptes amis perdus. Points clés :

  • sauvegarder PostgreSQL avant (la migration n’est pas réversible) ;
  • syn2mas check puis --dry-run s’exécutent sans arrêter Synapse ;
  • supprimer les registrations des ponts (/var/lib/mautrix-*/*-registration.yaml) pour qu’elles se régénèrent avec le flag MSC4190, puis redémarrer Synapse.

Inscrire des amis avec MAS

Section intitulée « Inscrire des amis avec MAS »

Les jetons d’inscription ne passent plus par l’API admin de Synapse : ils se créent sur l’hôte Matrix avec mas-cli (manage issue-user-registration-token), et la promotion d’un compte administrateur devient mas-cli manage promote-admin. La procédure détaillée sera documentée une fois l’outillage validé (voir l’encart ci-dessus).

Salons publics avec validation (knock)

Section intitulée « Salons publics avec validation (knock) »

Un salon public au sens Matrix se rejoint librement : pas de validation. Pour un salon « joignable mais sur approbation », utiliser la règle d’accès knock (toquer) — réglage par salon, côté client, sans configuration serveur.

  1. Dans Element : salon → Paramètres du salon → Sécurité & confidentialité.

  2. Accès au salon → « Demander à rejoindre » (knock).

  3. Un utilisateur (local ou distant fédéré) demande l’accès ; un modérateur approuve ou refuse.

Vraiment privé ?

Pour un salon confidentiel : le retirer de l’annuaire et le passer en knock ou sur invitation. Le chiffrement de bout en bout est déjà actif par défaut sur les nouveaux salons.

Diagnostiquer un bot muet

Section intitulée « Diagnostiquer un bot muet »

Vérifications côté client

Vérifier sur le client (Element, Fractal…) que le bot n’est pas ignoré. Faire un clic droit sur l’utilisateur correspondant au bot → information → sécurité, puis “ignoré”. Si le bot répond bien dans les logs mais que rien ne se passe, il se peut que le blocage soit côté client.

Un bot qui ne répond pas à help alors que son service tourne tombe presque toujours dans l’un des deux cas : ses tokens ne correspondent plus à la registration chargée par Synapse, ou il reçoit les messages mais ne peut pas les déchiffrer. Diagnostic dans l’ordre :

  1. Le service tourne-t-il vraiment ?

    S’il redémarre en boucle avec « The as_token was not accepted », l’enregistrement en mémoire de Synapse ne correspond plus aux tokens du pont (typique après un reset ou une migration, voir l’encart plus haut) : systemctl restart matrix-synapse, puis redémarrer le pont.

  2. Le message arrive-t-il au pont ? Suivre Synapse pendant l’envoi d’un help au bot :

    Une ligne PUT .../_matrix/app/v1/transactions/... 200 doit apparaître. Si rien ne part, l’appservice n’est pas chargé : vérifier la présence de l’enregistrement dans /var/lib/mautrix-<pont>/ et redémarrer Synapse.

  3. Le pont peut-il le déchiffrer ? Si la transaction passe (200) mais que le bot reste muet, suivre ses logs pendant un nouvel envoi :

    Chercher decrypt, session, verification, dropping. Un échec de déchiffrement signifie que la clé de session du salon n’a pas été partagée avec le device actuel du bot (cas classique après un reset du pont : son identité e2ee a changé).

  4. Recréer la conversation. Quitter le message direct avec le bot et en démarrer un nouveau : le client établit une session de chiffrement neuve et partage la clé avec le device actuel du bot. Vérifier au passage que sa propre session Element est vérifiée.

  5. Dernier recours : réinitialiser l’état du pont (encart ci-dessous), puis redémarrer Synapse pour recharger l’enregistrement.

Statuts WhatsApp désactivés

Le pont WhatsApp ne relaie pas les « statuts » (stories) : l’option network.enable_status_broadcast est forcée à false, sans quoi chaque utilisateur serait réinvité en boucle dans un salon « WhatsApp Status Broadcast ».

Réinitialiser un pont

Les données d’un pont (sessions, correspondance des salons) vivent dans /var/lib/mautrix-*. Supprimer ce répertoire déconnecte tous les utilisateurs du pont et casse la correspondance des salons existants : à ne faire qu’en dernier recours, service arrêté.
