Security rule catalog

This page summarizes which ANSSI rules are activated, according to the level and category of a host. It is the reference to consult before hardening a machine.

Understand first

The concepts used here (levels, categories, labels, MAC, egress…) are explained in Security notions.

Reminder: level and category

Section titled “Reminder: level and category”

Levels are cumulative: intermediary adds its rules to those of minimal, and so on. The category adds the rules specific to a workstation (client) or a server (server).

Recommended category per profile

Section titled “Recommended category per profile”
ProfileRecommended categoryWhy
minimal, portablebaseUniversal base
desktop, laptopclientScreen, USB, interactive session
server, gateway, hcs, vmserverExposed services, network

Today: base everywhere

Automatic profile → category mapping is not yet active. All hosts are set to minimal / base by default. To apply a category, set category manually on the relevant host.

R3, R5, C6: warning, no boot reconfiguration

From intermediary onward, these rules emit a warning when the protection is not detected, but do not reconfigure the boot. Secure Boot (lanzaboote, key enrollment) and LUKS encryption are declared in the host disk configuration (disko); the module only signals their absence. The future checkScript will verify their actual state.

R28, R52, R55: designated services and partitions

R28 hardens /tmp, /proc and /dev/shm by default; options on /var/log, /srv, /opt, /home are added via darkone.security.filesystem.extraMountHardening (separate partitions required). R52 (0750 mode for runtime directories) and R55 (private /tmp per service) apply to units listed in darkone.security.services.hardenedUnits — empty by default. See R63.

R63: opt-in sandboxing per service

Full systemd confinement (ProtectSystem=strict, system call filters, MemoryDenyWriteExecute, UMask=0027…) applies to units declared in darkone.security.services.hardenedUnits. Empty by default: blind hardening would break services. The list acts as a floor, overridable per unit, and needs-jit lifts MemoryDenyWriteExecute for JIT engines.

R61 : managed by DNF

Updates are centrally managed by the declarative deployment. The autoUpgrade mechanism is therefore disabled at all levels.

Tag kernel-recompile

The high-level kernel hardening (R15 to R27, C1) requires recompiling the kernel. On a machine without this need, add kernel-recompile to excludes to ignore it.

Adapt without breaking everything

Section titled “Adapt without breaking everything”

Two safety valves allow bypassing a specific protection without lowering the global level.

  • Tags (excludes): exclude a group of incompatible rules (e.g. needs-jit, no-ipv6).
  • Exceptions (exceptions): bypass one rule, justification required.

Details on these safety valves are in Security concepts.

Module under stabilization

Raise the level progressively and test each host before production.
