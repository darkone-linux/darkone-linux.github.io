Creating a "service" type module

A DNF “service” module wires a self-hosted piece of software (web UI, API, storage…) into the framework: reverse proxy, homepage, persistence, firewall, SSO and per-host activation. This page describes how to create and maintain such a module, from header to commit, using the oxicloud module as a running example.

Prerequisites

First read the service module standard (dnf/modules/service/AGENTS.md) and the code headers. Option details are generated in the reference.

Module anatomy

Section titled “Module anatomy”

A module lives in dnf/modules/service/<name>.nix. It is auto-imported via the generated default.nix (run just generate after creation).

Header

Section titled “Header”

Mandatory header comment: 1st descriptive line, then paragraphs and Starlight admonitions (note, tip, caution, danger) oriented toward usage / maintenance.

Options

Section titled “Options”

Imposed schema: darkone.service.<name>.<option>. Start with only the enable option; only add options for matters not resolved by the framework.

Not an option: service parameters

domain, global, title… are not module options: they are parameters declared in etc/config.yaml, resolved by the framework via dnfLib.extractServiceParams. See Per-host activation.

config structure

Section titled “config structure”

Always lib.mkMerge of two blocks:

  1. Unconditional block: registration with the service system and

  2. lib.mkIf cfg.enable block: the actual NixOS configuration (the upstream service, the firewall, the secrets…).

Injected arguments and consumer boundary

Section titled “Injected arguments and consumer boundary”

Beyond lib, config, pkgs, each service module receives specialArgs:

  • host: current host (hostname, zone, ip, vpnIp, services, profile…).
  • hosts: all hosts in the deployment (all zones).
  • zone: current host’s zone (name, domain, gateway, lang…).
  • network: global config (domain, coordination, services, zones).
  • dnfLib: framework helpers (never import dnf/lib/ manually).
  • dnfConfig: framework config registries (dnfConfig.network.ports.<key>: port registry).
  • workDir: consumer workspace root.
  • users: all users.

Framework ↔ consumer boundary

A framework module accesses consumer files (usr/secrets/…, usr/www/…) only via workDir + "/usr/…", never by relative path (./../../usr/…). The framework flake root is not the consumer workspace.

Registration and activation

Section titled “Registration and activation”

Service exposition

Section titled “Service exposition”

The unconditional block declares darkone.system.services.service.<name>, read by modules/system/services.nix to generate reverse proxy, homepage, persistence and backup.

  • proxy.servicePort: internal HTTP port; the Caddy reverse proxy connects to it
  • persist.{dirs,dbDirs,varDirs,mediaDirs,…}: persisted / backed-up paths.
  • defaultParams: default metadata (title, description, icon), overridable from config.yaml. The icon uses the selfh.st 🡕 slug (prefixed sh- by the framework).

Protocols served on the same HTTP port (WebDAV, CalDAV, CardDAV for OxiCloud) naturally go through the single Caddy vhost: nothing special to configure.

Network, database and secrets

Section titled “Network, database and secrets”

Internal ports

Section titled “Internal ports”

Any internal port a module binds, opens in the firewall or proxies must come from the central registry dnf/config/network.nix, read via dnfConfig.network.ports.<key> → never a hardcoded literal in the module. It is the single source of truth: the same number is never written twice and two services cannot silently conflict.

  1. Name the key <service>[Usage]: the ports. prefix already says “port”, so the key specifies what it is used for (kanidmReplication, garageRpc). A <service> alone designates the main port. Keep network.nix sorted by value.

  2. Add the value in ports.<key> when DNF itself binds the port, then read it in the module:

  3. Port inherited from upstream (config.services.<x>.port, not bound by DNF): register it in ports.reserved (anti-conflict denylist) instead; the module continues to read config.services.<x>.port.

Tested uniqueness

dnf/tests/unit/config/network_test.nix checks global uniqueness across all ports.* values and the ports.reserved list. Any conflict fails the unit test suite.

TLS outside HCS

Outside HCS, a DynamicUser service cannot read synced Caddy certificates. For a web service, rely on the automatic reverse proxy (proxy.servicePort) + HTTP in the clear on the trusted LAN/VPN; do not manage ACME manually.

Sops secrets

Section titled “Sops secrets”
  • Declare sops.secrets.<name> = { mode = "0400"; owner = "<svc-user>"; } (default file usr/secrets/secrets.yaml).
  • A DynamicUser = true service cannot own a sops secret. If it loads the key via LoadCredential (read as root), a root 0400 secret suffices; otherwise, fall back to a static group + SupplementaryGroups.

External binaries in systemd units

Every external binary in a systemd unit (script, ExecStart=, preStart…) must be written as a full store path: ${pkgs.<pkg>}/bin/<cmd> (systemd’s PATH is empty). Shell builtins (echo, [, for…) are exempt.

OIDC Authentication with Kanidm

Section titled “OIDC Authentication with Kanidm”

SSO wiring is automatic and conditional on the presence of an idm (Kanidm) service on the network (zone or global). Three ingredients:

  1. OAuth2 Template (unconditional block): darkone.service.idm.oauth2.<name>. Kanidm provisions it as an OAuth2 client when idm is enabled.
  2. OIDC Context: dnfLib.mkOidcContext → clientId, secret, idmUrl. idmUrl == null when no idm exists: all wiring is then cut off.
  3. Endpoints: dnfLib.mkKanidmEndpoints idmUrl clientId → issuerUrl, authUrl, tokenUrl, userinfoUrl…

Conditionality

The OAuth2 template remains declared unconditionally (harmless without idm). Only the service-side wiring (settings.oidc, secret, env) is guarded by lib.mkIf hasIdm, leaving upstream behavior intact in the absence of Kanidm. Reference patterns: outline.nix, immich.nix, forgejo.nix.

Protecting a service without OIDC

Section titled “Protecting a service without OIDC”

When the service cannot speak OIDC (landing page, static site, UI without auth), DNF protects its vhost upstream via oauth2-proxy + Caddy forward auth, backed by Kanidm. A single oauth2-proxy instance per gateway, shared by all protected services in the zone. Everything is declarative on the service record (unconditional block), via two proxy options:

OptionDefaultRole
proxy.isProtectedfalsePuts the vhost behind oauth2-proxy (forward auth)
proxy.allowedGroups[ ]Allowed Kanidm groups (empty: any authenticated user)

Anchor and prerequisites

The login flow (/oauth2/*) is hosted on the zone’s landing page FQDN: the homepage service must therefore be present as soon as a service is protected (an assertion checks this). Groups are compared by SPN (name@<domain>) : the framework appends the suffix, declaring the short name suffices.

Native OIDC or forward auth?

If the service can speak OIDC, prefer native wiring (previous section): it passes identity and roles to the service. Forward auth only guards the door in front of a service that ignores who connects. Admin overview: Authentication and identities.

Sourcing an upstream module

Section titled “Sourcing an upstream module”

When only the NixOS module is missing (package already in nixpkgs), source it from a fork carrying the PR, without rebuilding it.

  1. Add in dnf/flake.nix a dedicated input pointing to the fork branch, without inputs.nixpkgs.follows (we want the fork tree as-is):

  2. Inject the module path into the modules list of mkNode (dnf/lib/mk-configuration.nix) → parsed unconditionally, with no effect as long as services.<name>.enable = false:

Package overlay: only if needed

Distinguish two cases. OxiCloud: the package is already in nixpkgs → no overlay, pkgs.oxicloud comes from the main tree. GeneWeb: the package was missing (OCaml deps) → an overlay (lib/overlays/geneweb.nix) injects pkgs.geneweb from the fork.

Cleanup after merge

Cleanup once the PR is merged into nixos-unstable: remove the input, remove the module path (and any overlay). The wrapper modules/service/<name>.nix stays unchanged.

Finalize and commit

Section titled “Finalize and commit”
  1. just generate: regenerates default.nix (auto-import of the new module).
  2. just clean: fix + check + generate + format (mandatory before commit).
  3. Evaluate / build a host to validate, then commit.

Commit message: one line, 80 characters max, English, format <type>(<subject>): <message>, e.g. module(oxicloud): full DNF service with OIDC.
