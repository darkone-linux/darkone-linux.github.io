Security modules

Note

A security module hardens the system, its programs and functionalities.

🔑 darkone.security.complement

Section titled “🔑 darkone.security.complement”

Cross-cutting complementary measures — Annex A (C1–C12). (wip)

These measures have no ANSSI number but are essential to the guide’s spirit. Covers linux-hardened patches (C1), Lockdown LSM (C2), USBGuard (C3), nftables deny-by-default (C4), hardened SSH (C5), LUKS2 (C6), NTP/NTS (C7), secure DNS (C8), disabled core dumps (C9), PAM anti brute-force (C10), legal banners (C11), and cron/at restriction (C12).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

Lockdown LSM level (C2)

  • none: no Lockdown.
  • integrity: forbids kernel modifications via userspace.
  • confidentiality: same + forbids reading kernel secrets.

C2 — Lockdown LSM

lockdown=confidentiality forbids kexec, /dev/mem writes, MSR, hibernation. Breaks dmidecode on some zones, flashrom, i2c-tools.

C4 — Egress filtering

The egress deny-by-default policy breaks tools that reach CDNs (curl, kernel downloads). Required: a controlled outbound HTTP(S) proxy.

C5 — Hardened SSH

SSH tunnels, agent forwarding, and X11 disabled. Adapt admin workflows to ProxyJump (ssh -J). This module overrides core.nix for SSH.

  • enable bool Enable complementary ANSSI measures (C1–C12).
  • lockdownLevel enum [ "none" "integrity" "confidentiality" ] Lockdown LSM level (C2)
  • lsmStack listOf str LSM stack order (C2, R11, R20). Modifies boot.kernelParams lsm=…
  • ntpServers listOf str NTP/NTS servers (C7).
  • useNts bool Enable NTS (Network Time Security) for NTP authentication (C7).
  • sshBanner str SSH banner displayed before authentication (C11).
  • cronAllowedUsers listOf str Users allowed to schedule cron jobs (C12).
  • atAllowedUsers listOf str Users allowed to schedule at jobs (C12). Mirrors cronAllowedUsers by default.
  • egressAllowlist listOf str IPv4/IPv6 addresses or CIDRs allowed outbound under the C4 reinforced egress deny-by-default policy. Empty list keeps the policy but only loopback and established/related traffic pass (proxy required).

🔑 darkone.security.filesystem

Section titled “🔑 darkone.security.filesystem”

Partitioning, file tree, and file permissions (R28–R29, R50–R57). (wip)

Covers secure mount options (R28), restriction of /boot (R29), permissions of sensitive files (R50), passwords kept out of the store (R51), sockets (R52), orphan files (R53), sticky bit (R54), per-user temporary directories (R55), and setuid (R56, R57).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R28 — noexec /tmp

noexec on /tmp breaks many installers and compilers (pip, cargo, gcc). Workaround: export TMPDIR=$XDG_RUNTIME_DIR in shell profiles.

R29 — /boot noauto

noauto on /boot requires a manual remount on every NixOS update. A nixos-rebuild wrapper must ensure automatic mount/unmount.

  • enable bool Enable ANSSI filesystem hardening (R28–R29, R50–R57).
  • allowedSetuid listOf str Allowlist of tolerated setuid/setgid binaries (R56, R57).
  • extraMountHardening listOf str Mount points to harden with nosuid,nodev,noexec (R28). Each listed path MUST be a separate partition declared elsewhere (hardware- configuration / disko); the options are merged into its existing fileSystems entry. An explicit list (rather than a global flag) avoids guessing the layout and never creates phantom mounts. Do not list /home if users run scripts there — noexec would break them.

🔑 darkone.security.integrity

Section titled “🔑 darkone.security.integrity”

Filesystem integrity — HIDS (R76–R77). (wip)

Covers sealing and integrity verification via AIDE (R76) and protection of the sealed database with GPG signature plus remote copy (R77).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R76 — AIDE

A full scan can take 15–60 min depending on FS size and CPU. Schedule during off-peak hours. Frequent false positives on /etc/resolv.conf, /var/lib, /var/cache — add them to exclusions.

R77 — Signed database

The AIDE database must be re-signed at every major NixOS update (the store changes). Plan a baseline update procedure.

  • enable bool Enable ANSSI HIDS integrity — AIDE (R76–R77).
  • aideRemoteCopy nullOr ( submodule { options = { host = lib.mkOption { type = str; description = "Remote host for the AIDE database copy."; }; sshKeyFile = lib.mkOption { type = path; description = "Path to the private SSH key for the remote copy."; }; }; } ) Remote copy of the GPG-signed AIDE database (R77). null = disabled.
    • host str Remote host for the AIDE database copy.
    • sshKeyFile path Path to the private SSH key for the remote copy.

🔑 darkone.security.journaling

Section titled “🔑 darkone.security.journaling”

Logging and auditd (R71–R73). (wip)

Covers persistent logging with TLS forwarding (R71), per-service dedicated journals (R72), and auditd with ANSSI rules (R73).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R71 — Seal=yes

Seal=yes requires journalctl --setup-keys (out-of-band audit key). Without this key, journal integrity cannot be verified.

R73 — immutable auditd

The configuration audit.rules = [ "-e 2" ] makes auditd immutable: any rule change requires a reboot. Test in staging before deployment.

  • enable bool Enable ANSSI logging and auditd (R71–R73).

🔑 darkone.security.kernel-build

Section titled “🔑 darkone.security.kernel-build”

Static kernel configuration — requires recompilation (R15–R27). (wip)

These rules require a custom kernel via boot.kernelPackages. NixOS allows this through structuredExtraConfig. All these rules carry the kernel-recompile tag: they are skipped if that tag is in excludes.

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

No NixOS binary cache

Enabling these rules implies a local kernel recompilation (~30–60 min per update depending on CPU). No public binary cache covers a custom kernel. Reserve for fleets with proper MCO tooling.

Out-of-tree modules

R18 (signed modules) makes signing any external module mandatory (NVIDIA, ZFS, VirtualBox, v4l2loopback…) via the Nix pipeline.

  • enable bool Enable the ANSSI kernel recompilation module (R15–R27).

🔑 darkone.security.kernel-params

Section titled “🔑 darkone.security.kernel-params”

Dynamic kernel configuration: boot parameters and sysctls (R8–R14). (wip)

Rules applicable without recompiling the kernel. Covers memory options (R8), system sysctls (R9), disabling module loading (R10), Yama/ptrace (R11), IPv4 network sysctls (R12), IPv6 disabling (R13), and filesystem sysctls (R14).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R8 — SMT disabled

mds=full,nosmt and mitigations=auto,nosmt disable hyperthreading, which reduces multi-threaded throughput by around 30%. Evaluate on compute servers before enabling the intermediary level.

R9 — unprivileged_userns_clone=0

Breaks rootless Docker, rootless Podman, non-suid bubblewrap, and Flatpak. Use the R9 exception if these tools are required.

R10 — Modules disabled

Any new device requiring a module not pre-loaded needs a reboot. Mutually exclusive with nixos-rebuild switch on the driver side.

  • enable bool Enable ANSSI dynamic kernel parameters (R8–R14).

🔑 darkone.security.mac

Section titled “🔑 darkone.security.mac”

Mandatory Access Control — MAC (R37, R45–R49). (wip)

R37 is a meta rule: valid if at least R45 (AppArmor) or R46 (SELinux) is active. SELinux (R46–R49) is not supported on NixOS and is excluded by default via exceptions. AppArmor (R45) is partially supported.

Sandboxing

When MAC is unavailable, NixOS often relies on systemd options (systemd sandboxing) to isolate services.

NixOS and MAC

SELinux is structurally unsupported on NixOS (R46–R49 are exceptions by default). AppArmor is available but with few ready-to-use profiles. The absence of a profile for an exposed service is a false sense of security.

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

Tag no-mac

Use the no-mac tag in excludes to disable R37 and R45 with an explicit justification in exceptions.R37.rationale.

  • enable bool Enable ANSSI MAC module — AppArmor/SELinux (R37, R45–R49).

🔑 darkone.security.network

Section titled “🔑 darkone.security.network”

Network services — isolation and supervision (R78–R80). (wip)

Covers isolation of network services (R78: server category), hardening and monitoring of exposed services (R79: fail2ban, HTTP headers, ANSSI TLS) and reduction of the network surface (R80).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R79 — Strict CSP

Content-Security-Policy: default-src 'self' breaks tools without nonces (dashboards with CDN, third-party widgets). HSTS preload is irreversible — only set on a fully controlled domain.

R80 — Listener filtering

Some services (KDE Connect, mDNS) listen by default on all interfaces. Declare them in network.publicListeners or reconfigure them.

  • enable bool Enable ANSSI network hardening (R78–R80).
  • exposedServices listOf str Public services for which fail2ban and supervision will be enabled (R79).
  • publicListeners listOf str Services allowed to listen on 0.0.0.0 / :: (R80).
  • httpsHeaders bool Adds ANSSI HTTP security headers in Nginx/Caddy (R79).
  • tlsCiphers listOf str ANSSI TLS cipher suite for Nginx (R79).

🔑 darkone.security.packages

Section titled “🔑 darkone.security.packages”

Package management and updates (R58–R61). (wip)

Covers installing only what is strictly necessary (R58), trusted repositories (R59), hardened repositories (R60: linux_hardened), and regular updates (R61).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R59 — allow-import-from-derivation=false

Breaks some complex flakes (Haskell, heavy Python with Nix generators). Document exceptions in nix.settings.

R60 — linux_hardened

The linux_hardened kernel may lag a minor version behind nixpkgs-unstable. Third-party modules (NVIDIA, ZFS) are not guaranteed compatible.

  • enable bool Enable ANSSI package management (R58–R61).
  • trustedSubstituters listOf str Allowlist of authorized Nix binary caches (R59).
  • trustedPublicKeys listOf str Public keys for authorized binary caches (R59).
  • allowedUris listOf str Prefix allowlist of URIs fetchable during evaluation (R59). Restricts builtins.fetch* / flake inputs to internal mirrors. Empty list disables the restriction.
  • maxSystemPackages nullOr int Soft threshold on the number of system packages (R58). Emits an evaluation warning when exceeded; null disables the check.

🔑 darkone.security.preboot

Section titled “🔑 darkone.security.preboot”

Hardware configuration and secure boot (R1–R7). (wip)

Covers UEFI Secure Boot (R3), lanzaboote, bootloader password (R5), signed UKIs (R6), and IOMMU (R7). R1 and R2 (hardware/firmware) are out of NixOS scope and only produce a note in the report.

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R3/R4 — Secure Boot

The lanzaboote integration requires an initial physical key enrollment (sbctl enroll-keys). R4 (replacing Microsoft keys) carries a bricking risk if the machine does not allow restoration.

R7 — IOMMU

May cause crashes with some GPUs/Thunderbolt; I/O overhead ~5–15% on very high-throughput NVMe arrays.

  • enable bool Enable ANSSI secure boot — Secure Boot, IOMMU (R1–R7).

🔑 darkone.security.services

Section titled “🔑 darkone.security.services”

Hardening of systemd services (R62–R66). (wip)

Covers disabling unnecessary services (R62), reducing functionality via systemd security options (R63), privilege restriction (R64), isolation (R65), and hardening of containerization components (R66).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R63 — MemoryDenyWriteExecute

Breaks JIT runtimes (Java, V8, .NET, LuaJIT, Wasm). Use the needs-jit tag in excludes or exclude the service individually.

R66 — Docker userns-remap

Breaks host-to-container bind-mounts (UID shift). Migration of existing volumes required before enabling.

  • enable bool Enable ANSSI hardening of systemd services (R62–R66).
  • rootServicesAllowed listOf str systemd services allowed to run as root without CapabilityBoundingSet (R64).
  • hardenedUnits listOf str systemd units to which the ANSSI hardening baseline is applied (R52 RuntimeDirectoryMode, R55 PrivateTmp, R63 full sandbox via dnfLib.mkHardenedServiceConfig). Empty by default: blanket sandboxing can break services that need write access or extra syscalls, so units are opt-in. List framework-owned units known to tolerate confinement; per-unit serviceConfig overrides still win.

🔑 darkone.security.sudo

Section titled “🔑 darkone.security.sudo”

sudo hardening (R38–R44). (wip)

Covers the dedicated sudo group (R38), hardened sudo directives (R39), restriction of non-root targets (R40), NOEXEC limitation (R41), forbidding negations (R42), explicit argument specification (R43), and use of sudoedit (R44).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) are activated based on level, category, and excludes defined in darkone.system.security (via isActive).

R39 — requiretty

requiretty makes ssh user@host sudo cmd fail without an allocated TTY. The SSH -t option is required: ssh -t user@host sudo cmd.

R39 — rootpw

rootpw forces a root password shared between admins. Prefer targetpw or runaspw based on team policy.

  • enable bool Enable ANSSI sudo hardening (R38–R44).
  • allowedRootRules listOf str sudo rules allowed to target root (R40). Others must target a service account.

🔑 darkone.security.users

Section titled “🔑 darkone.security.users”

User accounts and authentication (R30–R36). (wip)

Covers unused accounts (R30), password policy (R31), inactivity locking (R32), admin accountability (R33), service accounts (R34), unique service accounts (R35), and umask (R36).

Activation

The enable option follows darkone.system.security.enable by default. Rules (Rxx/Cxx) activate according to level, category, and excludes defined in darkone.system.security (via isActive).

R30 — mutableUsers=false

NixOS no longer allows ad-hoc passwd/useradd. Any account change requires a full NixOS deployment (colmena apply).

R36 — umask 0077

Breaks group collaboration (/srv/share). Document that chmod g+rwx <file> is required for group sharing.

  • enable bool Enable ANSSI account management (R30–R36).
